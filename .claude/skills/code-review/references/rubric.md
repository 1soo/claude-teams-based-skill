# 카테고리별 감점 기준 (각 100점 만점)

각 카테고리는 100점에서 시작해 해당 항목이 발견될 때마다 점수를 뺀다(최저 0점). 90점 이상이면 통과.

## 1. 컨벤션 준수

담당 스택의 `implementation` skill `references/{역할}/conventions.md`, 설계 산출물(`docs/02_plan`)의 네이밍·구조와 얼마나 일치하는지 본다.

| 위반 항목 | 감점(건당) |
|---|---|
| `implementation` skill 컨벤션 문서(네이밍·디렉토리 구조·스타일 규칙) 위반 | -15 |
| 설계 산출물(`docs/02_plan`)과 다른 네이밍·구조 사용 | -10 |
| 새로 생성한 디렉토리에 `CLAUDE.md` 누락 | -10 |
| 프로젝트 내 기존 코드와 스타일(들여쓰기, 네이밍 케이스 등) 불일치 | -5 |

## 2. 재사용성

기존 컴포넌트·모듈·유틸을 두고 새로 작성했는지, 반복되는 로직을 공통화했는지 본다.

| 위반 항목 | 감점(건당) |
|---|---|
| 이미 존재하는 컴포넌트/함수/유틸로 대체 가능한데 새로 작성(중복 구현) | -20 |
| 동일·유사 로직이 3회 이상 반복되는데 공통 함수/컴포넌트로 추출하지 않음 | -15 |
| 기존 스택으로 충분한데 불필요한 신규 라이브러리·의존성 추가 | -10 |

## 3. 효율성

요청 범위를 넘는 구현, 불필요한 연산·호출, 남는 코드가 있는지 본다.

| 위반 항목 | 감점(건당) |
|---|---|
| 불필요하게 반복되는 API/DB 호출(N+1 패턴 등) | -20 |
| 설계·요구사항에 없는 범위까지 과잉 구현(YAGNI 위반) | -15 |
| 단발성 코드에 불필요한 추상화·레이어 추가 | -10 |
| 미사용 import·변수·죽은 코드 잔존 | -5 |

## 보고 형식

카테고리마다 아래처럼 정리한다.

```markdown
### {카테고리명} — {점수}/100 ({PASS/FAIL})
- {파일 경로}:{라인} — {위반 항목} (-{감점})
```

90점 미만 카테고리의 감점 사유만 담당 개발 에이전트에게 수정 요청으로 전달한다.
