# 코드 리뷰 (Standards축)

`dev-lead`가 도메인별 통합 테스트 통과 후, 커밋 전에 사용하는 컨벤션·품질 점검 skill. 요구사항·설계 충실도(Spec축)는 다루지 않는다 — 이 skill은 **컨벤션 준수·재사용성·효율성** 3개 카테고리만 채점한다.

## 채점 방식

- 대상: 해당 도메인의 diff(이번에 새로 작성·수정된 코드)만 본다. 기존 코드는 대상이 아니다.
- 카테고리마다 **100점에서 시작**해 [references/rubric.md](references/rubric.md)의 감점 항목에 해당할 때마다 명시된 점수를 뺀다(최저 0점).
- 카테고리별 **90점 이상 = 통과**, 90점 미만이면 그 카테고리의 감점 사유 전부가 "발견 사항"이다.
- 세 카테고리를 합산하거나 평균 내지 않는다. 각 카테고리를 독립적으로 판단한다.

## 절차

1. `implementation` skill의 담당 스택 `references/{역할}/conventions.md`를 확인해 이번 diff에 적용되는 컨벤션을 파악한다.
2. [references/rubric.md](references/rubric.md)의 감점 항목을 기준으로 diff를 훑으며 카테고리별 감점 사유를 기록한다.
3. 카테고리별 점수와 통과 여부를 정리해 보고한다(파일:라인 단위로 근거 명시).
4. 90점 미만 카테고리가 있으면 해당 발견 사항을 담당 개발 에이전트에게 `SendMessage`로 전달해 수정을 요청한다.
5. 수정 완료 후 **해당 카테고리만** 재점검한다(통과한 카테고리는 재점검하지 않는다). 모든 카테고리가 90점 이상이 될 때까지 반복한다.

## 공통 원칙

- 발견 사항은 반드시 구체적 파일:라인과 감점 사유를 함께 제시한다. 근거 없는 감점은 하지 않는다.
- 설계(`docs/02_plan`)·요구사항(`docs/01_analyze`) 충족 여부는 이 skill의 범위가 아니다(Spec축에서 별도로 다룸).
